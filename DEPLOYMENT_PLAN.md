# HandyToFile — план развёртывания

## Назначение

Этот документ разделяет требования для:

- машины разработчика
- машины для сборки релизного пакета
- рабочего ПК конечного пользователя

Для HandyToFile тяжёлый нативный тулчейн нужен только при разработке и сборке пакета. Обычный пользователь получает готовое приложение, а не репозиторий.

## 1. Машина разработчика

Используйте эту конфигурацию на машине, где ведётся разработка, сборка и тестирование форка.

### Необходимый тулчейн

- Rust через `rustup`
- Visual Studio 2022, набор инструментов C++
- Node.js LTS
- WebView2 Runtime
- LLVM с `libclang.dll`
- Vulkan SDK
- Git

### Необходимое окружение

- `cargo`, `rustc`, `rustup` доступны в `PATH`
- `node` и `npm` доступны в `PATH`
- `LIBCLANG_PATH` указывает на папку `bin` LLVM
- `VULKAN_SDK` указывает на корень Vulkan SDK

Типичные значения на Windows:

```powershell
$env:LIBCLANG_PATH = 'C:\Program Files\LLVM\bin'
$env:VULKAN_SDK = 'C:\VulkanSDK\1.4.341.1'
```

### Команды проверки

```powershell
cargo --version
rustc --version
node --version
& 'C:\Program Files\nodejs\npm.cmd' --version
Test-Path 'C:\Program Files\LLVM\bin\libclang.dll'
Test-Path "$env:VULKAN_SDK\Lib\vulkan-1.lib"
```

### Типичные проверки в процессе разработки

```powershell
cd C:\projects\HandyToFile
& 'C:\Program Files\nodejs\npm.cmd' install --ignore-scripts
& 'C:\Program Files\nodejs\npm.cmd' run build

cd C:\projects\HandyToFile\src-tauri
$env:LIBCLANG_PATH = 'C:\Program Files\LLVM\bin'
$env:VULKAN_SDK = 'C:\VulkanSDK\1.4.341.1'
cargo check
cargo test
```

### Требования к сети

Машина разработчика должна иметь доступ к:

- `https://index.crates.io`
- `https://crates.io`
- git-зависимостям, используемым Cargo
- эндпоинтам npm-реестра
- эндпоинтам для скачивания моделей приложением

Проверка доступности:

```powershell
Test-NetConnection index.crates.io -Port 443
Test-NetConnection crates.io -Port 443
Test-NetConnection registry.npmjs.org -Port 443

cd C:\projects\HandyToFile\src-tauri
cargo fetch
```

## 2. Машина для сборки релиза

Может совпадать с машиной разработчика.

### Необходимо

- все предварительные требования машины разработчика
- доступ к инструментам подписи, если будет применяться code signing
- достаточно чистое окружение для воспроизводимой сборки артефактов

### Типичные команды сборки релиза

```powershell
cd C:\projects\HandyToFile
& 'C:\Program Files\nodejs\npm.cmd' install --ignore-scripts
& 'C:\Program Files\nodejs\npm.cmd' run build

cd C:\projects\HandyToFile\src-tauri
$env:LIBCLANG_PATH = 'C:\Program Files\LLVM\bin'
$env:VULKAN_SDK = 'C:\VulkanSDK\1.4.341.1'
cargo test

cd C:\projects\HandyToFile
$env:LIBCLANG_PATH = 'C:\Program Files\LLVM\bin'
$env:VULKAN_SDK = 'C:\VulkanSDK\1.4.341.1'
& 'C:\Program Files\nodejs\npm.cmd' run tauri build
```

### Ожидаемый состав релизного пакета

- собранное десктопное приложение HandyToFile
- фронтенд-бандл, созданный Vite
- необходимые метаданные Tauri runtime
- поток загрузки моделей или встроенные модели, в зависимости от выбранной стратегии дистрибуции

## 3. Рабочее ПК конечного пользователя

На этой машине запускается только уже собранное приложение.

### Обычно требуется

- готовый релизный пакет HandyToFile
- WebView2 Runtime на Windows
- разрешение на использование микрофона
- разрешение на доступность / управление вводом, если требуется для автовставки

### Обычно не требуется

- Rust
- Node.js
- Visual Studio C++ build tools
- LLVM или `libclang`
- Vulkan SDK
- исходный код репозитория

### Чеклист первого запуска

1. Установить упакованное приложение.
2. Запустить его.
3. При запросе предоставить разрешение на микрофон.
4. При необходимости предоставить разрешение на доступность или управление вводом для автовставки.
5. Выбрать или скачать локальные модели распознавания речи.
6. Проверить дефолтный шорткат для вставки.
7. Проверить шорткаты захвата в Markdown-файлы.
8. При необходимости скорректировать пути к файлам заметок/задач/идей/покупок в настройках.

## 4. Рекомендации по упаковке форка

Для HandyToFile релиз должен сохранять оригинальный UX Handy и добавлять новый рабочий процесс захвата без раскрытия деталей, нужных только разработчику.

### Рекомендуемое поведение по умолчанию

- вставка в активное окно остаётся включённой по умолчанию
- Markdown-действия захвата включены по умолчанию
- пути для заметок, задач, идей и покупок редактируются в настройках
- профили распознавания и действия захвата предзаполняются в конфиге

### Рекомендуемые предположения о runtime

- целевому ПК не нужен никакой компилятор
- целевому ПК не нужны Cargo или npm
- целевому ПК не нужны LLVM или Vulkan SDK для обычной работы

Если в процессе упаковки окажется, что для конкретной модели или бэкенда на целевой ОС требуется специфическая runtime-библиотека — задокументировать это как предварительное требование к runtime. Не предполагать, что полный SDK нужен на пользовательских машинах.

## 5. Текущий статус верификации

Актуальная локальная верификация на Windows — успешна:

- `cargo check` проходит
- `cargo test` проходит
- `npm run build` проходит

Это подтверждает, что текущее состояние репозитория успешно собирается с необходимым dev-тулчейном.

## 6. Краткий итог

- Машина разработчика: полный нативный тулчейн обязателен.
- Машина для сборки релиза: то же самое плюс опциональная подпись.
- Рабочее ПК конечного пользователя: только готовый пакет приложения и стандартные runtime-разрешения.
